# GitHub 仓库分享审查 Skill

当用户准备将仓库公开分享、推广、或"拿高星标"时触发。涵盖从元数据到 README 结构到脱敏的全链路审查。

## 触发条件

用户说以下任一关键词时自动激活：
- "分享" "推广" "公开" "发布" "拿星标" "star"
- "/repo-share"
- 提到想让仓库被更多人看到

## 审查流程（5 步，逐项打勾）

### Step 1：仓库元数据（搜索曝光的第一入口）

```
□ 仓库名：是否包含目标关键词？（如 deepseek, claude-code）
□ 描述 (description)：是否在 200 字内讲清：目标用户 + 核心功能 + 独有亮点？
□ Topics：是否覆盖了用户搜索时会用的词？
  - 必选：技术栈关键词（如 deepseek, claude-code, mcp）
  - 必选：场景关键词（如 ai-coding, code-template, developer-tools）
  - 建议：平台关键词（如 windows, linux）
□ 主页 URL：是否有 homepage/website 链接？（可选但加分）
```

### Step 2：README 结构（路人 3 秒判断是否值得 star）

```
□ ⚠️ 置顶警示框：明确"本仓库做什么、不做什么"（防止错误预期）
□ 三层架构图：本仓库 / CLI工具 / API后端，标注每层谁负责
□ 适用人群：✅ 适合 / ❌ 不适合（过滤无效流量）
□ 核心问题列表：每个有 根因 + 解决 + 数据
□ 前置配置：如果依赖外部配置（如 API Key），折叠块放完整可复制代码
□ 对比表：用本仓库 vs 不用（或 vs 通用模板），差异化一目了然
□ FAQ：至少 5 条，覆盖最常见的误解和疑问
□ 设计决策：解释"为什么不那样做"（增加可信度）
□ 路线图：标注已完成 [x] 和计划中 [ ]
□ 中英双语：中英文 README 结构一致、内容同步
```

### Step 3：功能演示（证明不是空壳）

```
□ 截图：至少 2 张实际运行截图
□ 一行命令：提供 curl/wget 一行安装/更新命令
□ 交互式脚本：如有 init.sh，在 README 显著位置展示用法
```

### Step 4：脱敏审查（公开仓库 CRITICAL）

```
□ 无个人文件路径 (C:\Users\xxx, /home/xxx)
□ 无 API Key / Token / Secret / Password
□ 无内部 IP 地址 (192.168.x.x, 10.x.x.x, 172.16-31.x.x)
□ 无个人身份信息（学号、真名、邮箱——除非用户主动选择公开）
□ .gitignore 已排除 settings.local.json, .env, *.backup
□ 历史 commit 中无敏感信息残留（如有，需 squash/force-push 清理）
```

### Step 5：分享准备

```
□ 生成一段 3 句话的推广文案（可直接复制发帖）
□ 生成分享链接清单（仓库主页 + 一键安装命令 + 亮点直达链接）
□ 确认 PR 已合并、CI 通过、无 open issue 遗漏
```

## 审查后输出格式

每步完成后输出打勾状态 + 未通过项的修复建议。最后输出汇总：

```
## 审查完成

- 元数据：n/m 通过
- README：n/m 通过
- 功能演示：n/m 通过
- 脱敏：n/m 通过
- 分享准备：n/m 通过

🔒 脱敏声明：已确认 [repo-name] 为公开仓库，以上审查清单全部通过。
```

## 注意事项

- 脱敏审查（Step 4）必须逐条确认，不可跳过。发现敏感信息立即停止并告知用户。
- README 结构审查时，不强制要求每一项都有——根据仓库规模和目标判断。小型工具仓库可以精简。
- 分享文案只生成模板，不自动发布到任何平台。
- 如果仓库是私有的，提醒用户确认是否要改为公开，改之前的脱敏审查加倍严格。
